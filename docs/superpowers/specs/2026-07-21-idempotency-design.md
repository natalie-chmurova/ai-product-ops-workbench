# Idempotency for the meeting → ClickUp sync — Design

**Date:** 2026-07-21
**Status:** Approved (design), pending implementation
**Scope:** Tier 2, step 2 of the AI Product Ops Workbench

## Problem

The `Meeting → ClickUp (sync: create-or-update)` workflow applies decisions in four
places: `Add comment (auto)`, `Create task (auto)`, `Add comment (appr)`,
`Create task (appr)`. It has **no deterministic memory** that a given effect was
already applied. Consequences of re-processing the same meeting:

- Re-running the same transcript re-adds the same comments → the board fills up with
  duplicate comments (the semantic `Match agent` partially prevents duplicate *tasks*
  by choosing UPDATE when the task already exists, but it does **not** stop duplicate
  comments).
- The same status item repeated across weekly syncs produces the same comment twice.

## Goal / non-goals

**Goal.** Re-processing the same meeting (or the same item across meetings) must not
produce duplicate side effects. Concretely: applying the *same effect* twice is a
no-op the second time.

**Scenarios covered (user chose "both"):**
- **A — exact re-run:** the same transcript is processed again (workflow retry,
  accidental re-run, duplicate webhook).
- **B — repeated items across meetings:** a follow-up sync restates already-discussed
  items; we don't want a second identical comment on the same task.

**Non-goals.**
- Semantic de-duplication of *rephrased* items — that is the `Match agent`'s job
  (it merges "same task, different words"). The idempotency key is deterministic and
  only catches exact/near-exact repeats. See "Two layers" below.
- No Telegram notification on a caught duplicate (a log entry + end-of-run summary is
  enough — decided during design).

## Key decisions

### The unit is an *effect*, not a meeting or an item

Idempotency is keyed on the **applied effect**, computed **from the final effect —
after the `Match agent` decision, before applying**. This makes the guard independent
of what the agent decided, and covers both scenarios:

- re-run → same items → same effect keys → suppressed;
- same item text in another meeting → same effect key → suppressed;
- a legitimately new update (different wording / progress) → different key → passes.

Two kinds of effect:

| Effect | Key payload | Notes |
|---|---|---|
| **UPDATE** (comment on a task) | `"UPDATE\|" + target_task_id + "\|" + normalize(comment)` | `target_task_id` used as-is (already unique) |
| **NEW** (create a task) | `"NEW\|" + normalize(title + " " + description)` | no task id exists yet |

`key = sha1(payload encoded as UTF-8).hexdigest()`.

### Normalization spec (must be byte-identical in Python and n8n JS)

Order is fixed: **lowercase → collapse whitespace → trim.**

- Python: `re.sub(r"\s+", " ", text.lower()).strip()`
- JS (n8n): `text.toLowerCase().replace(/\s+/g, " ").trim()`

`\s+` collapses any run of spaces/tabs/newlines to a single space. The key is the
SHA-1 hex digest of the UTF-8 bytes of the payload string.

Because the store is shared between the Python module and the n8n Code node, the two
implementations must produce the *same* key for the *same* input. This is enforced
with **reference vectors** (`tests/idempotency_vectors.json`): input → expected sha1,
generated by the Python reference implementation and asserted by both the pytest suite
and a manual check of the n8n node.

### Store — local append-only JSONL

`state/processed_effects.jsonl`, next to the existing `decision_log.jsonl` mechanics
(same fs Code node style). One record per applied effect:

```json
{"key": "<sha1>", "kind": "UPDATE", "target": "86eyay0p7", "preview": "saved cards unblocked", "ts": "2026-07-21T…Z", "source": "transcript_followup"}
```

Loaded into an in-memory `set` of keys on start → O(1) membership check.

The store is **runtime state, not an artifact**: real `state/processed_effects.jsonl`
is git-ignored; a small `state/processed_effects.example.jsonl` is committed so the
record shape is visible.

### Three correctness rules

1. **Check before applying, write the key after success.** If create/comment fails,
   the key is *not* written, so a failure never permanently silences an effect.
   (At-least-once delivery + dedup-on-success = effectively exactly one successful
   effect.)
2. **In-run dedup.** Two identical effects within a single run are marked in one pass
   *before* anything is applied. Because `commit` writes a key only *after* a successful
   effect (later in the flow), the store can't yet reflect the first one — so the
   marking pass keeps an in-memory `seen_this_run` set and suppresses the second
   immediately.
3. **Don't go silent.** A caught duplicate is logged as `skipped_duplicate` and the
   run returns a summary `applied: N, skipped: M`. Visibility matters — a human should
   see the board was protected, not "nothing happened".

## Architecture

### Python module `src/idempotency.py` (reusable core)

The core lives in Python (testable via pytest, demonstrable via CLI) and is mirrored
into an n8n Code node — the project's established pattern (parsers and evals are built
this way).

```
class IdempotencyStore:
    __init__(path)                 # loads existing keys into a set
    is_duplicate(key) -> bool      # store membership OR seen this run
    commit(key, meta)              # append to JSONL; call ONLY after a successful effect

effect_key(kind, target, content) -> str   # sha1 hex; kind ∈ {"UPDATE", "NEW"}
normalize(text) -> str                      # lower → collapse ws → trim
```

`is_duplicate` checks both the loaded store and an internal `seen_this_run` set;
`commit` adds to both and appends the JSONL record.

### n8n insertion (3 nodes) into `meeting-to-clickup-sync.workflow.json`

1. **`Idempotency check`** (Code, fs-read) — right after `Parse decision`. For each
   effect computes `idem_key`, sets `is_duplicate` (store + in-batch dedup), passes the
   key through. Placed **before** `Confident?` so a duplicate never spends a Telegram
   approval.
2. **`Fresh only`** (Filter) — `is_duplicate=true` routes to the log as
   `skipped_duplicate`; fresh items continue into the existing `Confident?` →
   auto/HITL logic unchanged.
3. **`Mark processed`** (Code, fs-append) — on the paths **after a successful apply**
   (auto + approved), NOT on the Reject path. Appends the key to the store. Its
   placement after the apply nodes is what guarantees "key after success". Reject does
   not commit, so a later run will re-ask.

## Testing / proof

- **`tests/test_idempotency.py` (pytest):** normalization; key stability against the
  reference vectors; `is_duplicate`; commit-only-after-success; in-run dedup; the core
  invariant — **run a set of effects twice → the second run has `applied == 0`.**
- **CLI demonstration:** apply the effects from `transcript_followup` twice in a row →
  `applied: 6, skipped: 0` then `applied: 0, skipped: 6`. One-command proof, no n8n.
- **n8n end-to-end:** `execute` the sync workflow twice (auto branch, threshold −1) →
  in ClickUp the second run produces **0 duplicate tasks and 0 duplicate comments on
  repeated (UPDATE) items**. Caveat (not a bug): an item that was NEW in run 1 now
  exists, so the Match agent picks UPDATE and posts *one* comment on it — its effect key
  changed NEW→UPDATE, so the store doesn't link the two runs; from run 3 on that comment
  is suppressed too. Update the sanitized workflow copy in `n8n/`.
- **Packaging:** an "Idempotency (two layers of protection)" section in
  `docs/n8n-case.html` + the Notion spec, each with a "🗣 how to say it in an
  interview" note.

This is a **property test of the code** (a deterministic guarantee), **not** a fourth
eval — evals measure model quality (recall / decision accuracy) and idempotency does
not involve the model. It is deliberately not called an eval; the project keeps its
wording precise.

## Two layers (honest boundary)

The system covers both user scenarios, but with **two different mechanisms**:

- **Deterministic idempotency key** — catches the *exact / verbatim* repeat of an
  effect. Fully closes scenario A and verbatim B.
- **Semantic `Match agent`** — catches the *rephrased* repeat (same item, different
  words, in another meeting) by choosing UPDATE over a duplicate task.
- **Re-run nuance (NEW → UPDATE).** On a re-run, an item that was NEW the first time
  already exists as a task, so the Match agent switches it to UPDATE — its effect key
  changes, which is why the store doesn't tie the two runs together. Net answer to
  "what if I run it twice?": no duplicate tasks, no duplicate comments on status items,
  and one comment on each brand-new task the second time (stable from run 3 on).

Stating this prevents the false expectation that a single hash "understands meaning",
and it strengthens the story (two complementary guards) rather than weakening it.

## e2e finding: non-deterministic extraction (verified 2026-07-22)

Running the full n8n workflow twice does NOT reduce to "0 duplicates", and that is
expected. The board-sync workflow re-runs Extract + Match on every execution, and
claude-sonnet-5 is non-deterministic — its `temperature` is not configurable (the node
rejects it as deprecated for this model). So a re-run rephrases items: different titles,
different descriptions, even a different item count (8 vs 6 across runs), which changes
the content hash. Verified in the live runs — the stable item
`[QA] Additional Android test devices approved` produced the identical key
`4eeb0afc…` across three runs (correctly suppressible), while rephrased items drift to
new keys.

The honest boundary this establishes:
- The guard deterministically suppresses a repeat of an *identical* effect — exactly
  the real-world retry case: the same already-extracted payload delivered twice
  (duplicate webhook, workflow retry, re-delivery). Proven cleanly by the Python demo
  (double-run on a fixed set = 0 applied).
- A full re-run through a non-deterministic LLM extractor is a *different* problem: the
  input itself changes each time. Full coverage there needs a deterministic extractor,
  or idempotency keyed on a stable external `meeting_id + item_id` rather than on
  generated content.

What the live n8n run DID verify: the mechanism is wired and works end-to-end —
`Idempotency check` computes keys (crypto sha1), `Log decision` records a `duplicate`
flag, `Mark processed` writes the store via fs **only after a successful apply** (the
store matched the run's fresh effects), and `Fresh only` filters. The Mark-processed
node reads the marked items from `$('Idempotency check').all()` rather than the post-apply
`$json`, because the ClickUp apply node overwrites `$json` with its API response.
